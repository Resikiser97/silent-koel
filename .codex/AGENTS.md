## 你的職責範圍

### ✅ 你可以做的事

**靜態分析：**
- 讀取任何 JS 檔案進行分析
- 產出 `docs/dependency_map.md` 依賴地圖
- 驗證 import/export 語法正確性
- 檢查 import 路徑是否指向實際存在的檔案
- 檢查循環依賴
- 產出分析報告

**進度管理：**
- 更新 `docs/ESM_PROGRESS.md`（只更新進度狀態，不修改計劃內容）
- 在事件紀錄加入你的操作記錄

### ❌ 你絕對不能做的事

- 修改任何遊戲邏輯檔案（`systems/*.js`、`main.js`）的業務邏輯
- 修改任何 `config/` 下的數值
- 修改 `index.html` 的結構（除非 ESM 遷移任務明確要求）
- 自動修復你發現的問題（除非任務明確要求「修復」）
- 執行任何 git 操作（commit、push、merge）
- 刪除任何檔案

**發現問題時：只報告，不自動修復，等開發者確認。**

## 任務執行標準

### 依賴審計任務（Stage 1）
輸出必須包含：
1. 每個 JS 檔案定義了哪些全域變數/函式/常數
2. 每個 JS 檔案使用了哪些來自其他檔案的變數/函式/常數
3. 依賴關係中是否有循環
4. 建議的 ESM import 順序

格式：`docs/dependency_map.md`（Markdown 表格）

### 靜態語法檢查任務（Stage 2 各批次）
輸出必須包含：
1. 每個檔案的檢查結果（PASS / FAIL）
2. FAIL 的具體原因（哪一行、什麼問題）
3. 建議的修正方式（不要自己修）
4. 整體通過率

格式：在對話中直接輸出報告，同時更新 `docs/ESM_PROGRESS.md`

## 這個專案的特殊注意事項

1. **這個專案沒有測試框架**（沒有 Jest、Mocha 等），不要嘗試執行測試
2. **遊戲必須在瀏覽器裡測試**，你的沙盒環境無法模擬瀏覽器行為
3. **`gameState` 是全域狀態物件**，被幾乎所有 systems/ 讀寫，是最複雜的依賴節點
4. **`lang.js` 的 `t()` 函式**被所有 UI 相關檔案呼叫，必須在其他 UI 檔案之前載入
5. **`config/` 都是純資料常數**，沒有副作用，最適合先做遷移
6. **`index.html` 最後才動**，等所有 JS 檔案都完成 ESM 遷移後，再一次性修改載入方式
